# Аналіз життєвого циклу контейнера
## Варіант: A — Аналіз життєвого циклу контейнера

### 1. Запуск контейнера

Для виконання завдання я використав простий Python HTTP-сервер у Docker-контейнері.

Спочатку видалив старий контейнер, якщо він уже існував:

```bash
docker rm -f module7_app 2>/dev/null || true
```

Після цього запустив контейнер:

```bash
docker run -d \
  --name module7_app \
  -p 8080:8000 \
  python:3.12-slim \
  python -m http.server 8000 --bind 0.0.0.0
```

Перевірив, що контейнер запущений:

```bash
docker ps
```

Приклад результату:


```
CONTAINER ID   IMAGE              COMMAND                  PORTS                    NAMES
abc123def456   python:3.12-slim   "python -m http.se…"     0.0.0.0:8080->8000/tcp   module7_app
```

Також перевірив доступність сервісу з хоста:

```bash
curl http://localhost:8080
```

У відповідь сервер повернув HTML-сторінку зі списком файлів директорії.

### 2. Аналіз процесу всередині контейнера

Щоб подивитися, який процес є головним усередині контейнера, я виконав команду:

```bash
docker exec module7_app sh -c 'tr "\0" " " < /proc/1/cmdline; echo'
```

Результат:

```bash
python -m http.server 8000 --bind 0.0.0.0
```

Також я переглянув інформацію про процес через `/proc/1/status`:

```bash
docker exec module7_app sh -c 'cat /proc/1/status | head'
```

Приклад результату:
```
Name:   python
Umask:  0022
State:  S (sleeping)
Tgid:   1
Ngid:   0
Pid:    1
PPid:   0
```

#### Пояснення

Усередині контейнера процес `python -m http.server` має PID 1.
Це відбувається тому, що саме ця команда була передана як основна команда контейнера під час запуску через `docker run`

Контейнер — це не окрема віртуальна машина, а ізольований Linux-процес.
Життя контейнера напряму залежить від головного процесу з PID 1. Якщо цей процес завершується, контейнер також завершується.

### 3. Перевірка логів контейнера

Щоб згенерувати лог, я звернувся до сервера:

```bash
curl http://localhost:8080
```

Після цього переглянув логи контейнера:

```bash
docker logs module7_app
```

Приклад результату:
```
Serving HTTP on 0.0.0.0 port 8000 (http://0.0.0.0:8000/) ...
172.17.0.1 - - [13/Jun/2026 09:20:15] "GET / HTTP/1.1" 200 -
```

#### Пояснення

Логи, які показує команда `docker logs`, беруться зі стандартних потоків контейнера:

`stdout` — стандартний вивід;
`stderr` — стандартний потік помилок.

Docker підключає ці потоки до своєї системи логування.
Тому все, що застосунок пише в консоль, можна побачити через `docker logs`

У цьому випадку Python HTTP-сервер записує інформацію про запуск сервера та HTTP-запити в стандартний вивід або потік помилок, а Docker зберігає ці повідомлення.

### 4. Завершення контейнера

Для зупинки контейнера я використав команду:

```bash
docker stop module7_app
```

Після цього перевірив стан контейнера:

```bash
docker ps -a --filter name=module7_app
```

Приклад результату:

```
CONTAINER ID   IMAGE              STATUS                      NAMES
abc123def456   python:3.12-slim   Exited (0)                  module7_app
```

#### Пояснення

Команда `docker stop` спочатку надсилає головному процесу контейнера сигнал `SIGTERM`

`SIGTERM` — це сигнал ввічливого завершення процесу.
Він означає: “заверши роботу коректно”.

Якщо процес не завершується після `SIGTERM`, Docker чекає певний час, за замовчуванням приблизно 10 секунд, а потім надсилає сигнал `SIGKILL`

`SIGKILL` уже не можна проігнорувати. Він примусово завершує процес.

У моєму випадку Python HTTP-сервер коректно завершився після `docker stop`, тому контейнер перейшов у стан `Exited`

### 5. Висновок

У цьому завданні я побачив, що контейнер — це не магія і не повноцінна віртуальна машина, а ізольований Linux-процес.

Головний процес контейнера має PID 1.
Саме він визначає життєвий цикл контейнера:

якщо процес працює — контейнер працює;
якщо процес завершився — контейнер також завершується.

Docker дозволяє керувати цим процесом через сигнали.
Команда `docker stop` надсилає `SIGTERM`, а якщо процес не реагує, Docker може завершити його примусово через `SIGKILL`

Також я перевірив, що логи контейнера беруться зі стандартних потоків `stdout` і `stderr`, тобто з того, що застосунок виводить у консоль.

Таким чином, контейнер можна аналізувати як звичайний Linux-процес у контрольованому ізольованому середовищі.
